# SLOs and error budgets for non-deterministic systems

*Draft — gopherguard milestone M3. Content draft for engineer-to-engineer publication; unverified claims are marked inline with `<!-- VERIFY -->`.*

Classic SRE practice puts an SLO on availability: the service was either up and answered correctly, or it wasn't. That model quietly assumes determinism — given the same request, a correctly-deployed service gives the same correct response every time, so "success" and "failure" are unambiguous. Agents break that assumption at the root. An agent can be fully "up" — process reachable, model responding, no exceptions thrown — and still be wrong: it hallucinated a fact, took a needlessly expensive path, ran an unbounded loop, or made a judgment call a human would have overruled. None of that shows up as a 5xx. So what do you actually put an SLO on, if "did it return 200" tells you almost nothing about whether the system did its job?

## SLOs need to move from "did it respond" to "was the response good, cheap, and bounded"

The answer isn't to give up on SLOs for agents, it's to pick signals that are measurable and meaningful even though the underlying process is probabilistic. gopherguard's M3 telemetry schema (`trust.*`, `agent.hop`, `model.route_reason`, `mem.provenance` — the same trust-boundary vocabulary the detection rules query) already produces the raw material for several of these without adding new instrumentation:

- **Task success rate** — the fraction of sessions whose output matches a golden output or passes a grading rubric. This is the closest analogue to the old "did it work," except "worked" is now a judged property (exact match, rubric score above a threshold, or a grader model's verdict <!-- VERIFY --> rather than a status code, and the SLO target has to be set with that fuzziness in mind.
- **Hallucination-flag rate** — the fraction of sessions where an unsupported or ungrounded claim was flagged, whether by an automated grounding check or a human reviewer. You're not claiming zero hallucinations are achievable; you're bounding how often they're allowed to slip through ungraded.
- **Cost per session** — token and tool-call spend per completed task. Unlike a deterministic service where cost is roughly fixed per request, an agent's cost varies with how much reasoning, retrying, and tool-calling it does to reach an answer — so cost itself is a signal of the system drifting, not just a billing line item.
- **Loop-budget consumption** — how close sessions run to the loop/retry ceiling before being cut off. This ties directly to `GG-DET-03` (loop/cost runaway): the same "same span repeats more than five times" signal that fires a security detection also feeds an SLO as a *rate*, not a boolean. A session that used 4 of 5 budgeted iterations isn't a detection hit, but a fleet where the median session uses 4 of 5 is a system trending toward the failure mode the detection exists to catch.
- **HITL-denial rate** — the fraction of human-in-the-loop checkpoints that come back `denied` or `bypassed` rather than `approved` (`trust.hitl_result`). A rising denial rate means the agent is increasingly proposing actions a human won't sign off on — a leading indicator of drift in what the agent thinks is appropriate, well before it causes a security incident.

The throughline: every one of these is a rate computed over the same trust-boundary attributes that back the detection rules. The detections ask "did this bad sequence happen in this trace." The SLOs ask "at what rate does this bad sequence happen across the fleet." Same schema, two aggregations, one telemetry investment.

## Error budgets for probabilistic systems

The classic error-budget move — "we're allowed 43 minutes of downtime this month, spend it on risk" — still works here, but the unit of "error" changes. You're not budgeting for a fixed number of outages; you're budgeting for a *rate* of bad outcomes that a probabilistic system will produce no matter how well you build it, because "well built" doesn't mean "deterministic." A model-backed agent will hallucinate sometimes, will occasionally take a scope-widening path a reviewer would reject, will sometimes cost more than the median session. Zero is not an achievable target for any of these, and treating it as one just produces an SLO nobody believes and everybody ignores.

So the budget is explicit about the rate you tolerate: "hallucination-flag rate stays under budget-X <!-- VERIFY -->," "HITL-denial rate stays under budget-Y <!-- VERIFY -->," "no more than budget-Z percent of sessions exhaust the loop budget <!-- VERIFY -->." Where those specific thresholds land is a product and risk decision gopherguard hasn't fixed yet — the point of this milestone is building the measurement, not asserting numbers that haven't been earned by production data. What the trust telemetry buys you is the ability to *compute the current rate honestly*, which is the precondition for setting any threshold. You can't budget for a rate you can't measure, and you can't measure "how often did the agent do something a human would object to" without `trust.hitl_result` already stamped on every checkpoint.

This is also where the error-budget framing earns its keep over a plain dashboard: a budget forces the conversation from "is this number bad" to "how much of this are we willing to spend, and on what." A team that's burned most of its loop-budget-consumption allowance for the month should ask whether a recent prompt or tool change made the agent chattier — the same way an SRE team burning its downtime budget asks whether a risky deploy is worth the remaining slack. Non-determinism doesn't remove that conversation; it just changes which numbers are on the table.

## Making the non-determinism observable and boundable

None of this is useful if it lives in a test report nobody reads. The intent is a Grafana SLO dashboard built directly on the same trace store the detection rules query (Tempo/ClickHouse — see the M3 detections work), so the same trust-boundary attributes back both the security detections and the reliability SLOs side by side. A loop-budget-consumption panel next to a GG-DET-03 firing count makes a point that's easy to miss when SRE and AppSec run separate tooling: the same underlying behavior (a session running away on retries) is simultaneously a reliability problem (cost, latency, budget burn) and a security-relevant one (potential runaway/DoS-style abuse). One schema, one dashboard, both lenses.

That observability is also the reason this matters beyond a nice dashboard. These SLOs are a forward reference to M4: once task success rate, hallucination-flag rate, cost per session, loop-budget consumption, and HITL-denial rate are measured and budgeted, they become gates a CI/CD pipeline can check before a change ships — not just "did the unit tests pass" but "did this change regress the fleet's measured reliability and safety rates." That's the same trajectory error budgets took in classical SRE: first you measure, then you set a budget, then the budget starts blocking releases that would spend it too fast. Non-deterministic systems don't get a pass on that discipline; they need it more, because "it passed the tests" says less about a model-backed agent than it ever said about deterministic code.
